- Скрипт `postgres-setup.sh` создает fs на диске, монтирует его, создает нужный докер контейнер postgreSQL и выводит на экран номер сектора, необходимый для воспроизведения бага.

- В скрипте версия образа postgreSQL указывается в переменной PG_V и по-умолчанию равна 9.6.10. На этой версии баг еще существует.

- При работе в консоли psql данные записываются в оперативную память, чтоб их записать на диск необходимо вызвать команду `checkpoint`. При вызове этой команды вызывается функция ядра fsync(), которая переносит "грязные" страницы (области памяти с несохраненными на диск данными) на диск.

- Баг заключается в том, что если при fsync() произойдет ошибка, то ядро сотрет информацию с "грязных" страниц, которые ему не удалось сохранить, и пометит их как "чистые" страницы. При этом postgres, видя что страницы стали "чистыми", считает, что все данные успешно сохранились.

- Скрипт `postgres-setup.sh` создает контейнер `pg-test`.
    - Подключаемся к консоли psql 
    ```bash
    docker exec -it pg-test psql -U postgres
    ```
    - Настраиваем модель так, чтоб она возвращала Hardware Failure (Sense key: 0x04, ASC: 0x44, ASCQ: 0x00) при запросах к сектору, номер которого вернул скрипт `postgres-setup.sh`
    - В psql добавляем запись в таблицу
    ```sql
    INSERT INTO t(data) VALUES('WILL-BE-DELETED');
    ```
    - Пытаемся сохранить данные (сейчас мы должны наткнуться на ошибку, но все равно остаться в консоли psql)
    ```sql
    CHECKPOINT;
    ```
    - Сохраняем данные повторно (в этот раз должно завершиться успешно)
    ```sql
    CHECKPOINT;
    ```
    - В логах модели можно увидеть, что к указанному сектору был выполнен лишь один запрос на запись (тот, который вернул ошибку). При второй, якобы успешной, попытке запрос на запись на сектор даже не вызывается.
    - ***Настраиваем модель так, чтоб ошибки больше не было***
    - Выходим из контейнера с помощью `Ctrl-D`, чистим кэш и перезапускаем контейнер
    ```bash
    sudo sysctl -w vm.drop_caches=3 && docker restart pg-test
    ```
    - Подключаемся к консоли psql 
    ```bash
    docker exec -it pg-test psql -U postgres
    ```
    - Смотрим записи в таблице, записи `WILL_BE_DELETED` там быть не должно
    ```sql
    SELECT * FROM t;
    ```
    
- В версии 9.6.12 этот баг пофиксили, теперь если postgres видит что fsync() вернул ошибку, то он впадает в режим PANIC.

- Необходимо поменять в скрипте значение переменной PG_V на "9.6.12" и перезапустить скрипт.

- Скрипт создает контейнер `pg-test` с новой версией postgres.
    - Подключаемся к консоли psql 
    ```bash
    docker exec -it pg-test psql -U postgres
    ```
    - Настраиваем модель так, чтоб она возвращала Hardware Failure (Sense key: 0x04, ASC: 0x44, ASCQ: 0x00) при запросах к сектору, номер которого вернул скрипт `postgres-setup.sh`
    - В psql добавляем запись в таблицу
    ```sql
    INSERT INTO t(data) VALUES('WILL-BE-DELETED');
    ```
    - Пытаемся сохранить данные
    ```sql
    CHECKPOINT;
    ```
    - postgres должен впасть в PANIC, а контейнер `pg-test` завершить свою работу из-за ошибки
    - ***Настраиваем модель так, чтоб ошибки больше не было***
    - Чистим кэш и перезапускаем контейнер
    ```bash
    sudo sysctl -w vm.drop_caches=3 && docker restart pg-test
    ```
    - Подключаемся к консоли psql 
    ```bash
    docker exec -it pg-test psql -U postgres
    ```
    - Смотрим записи в таблице, запись `WILL_BE_DELETED` должна присутствовать
    ```sql
    SELECT * FROM t;
    ```
- Во втором случае запись `WILL_BE_DELETED` присутствует в БД, потому что при перезапуске контейнера происходит восстановление БД после PANIC, в этот момент запись и возвращается в БД.
